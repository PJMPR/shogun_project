\section{Cel dydaktyczny}

Celem przedmiotu jest nabycie praktycznych umiejętności projektowania i implementowania nowoczesnych aplikacji internetowych z podziałem na backend i frontend, w oparciu o Domain-Driven Design (DDD) oraz architekturę oktogonalną. Student poznaje podstawy budowy rozwiązań typu SOA, mechanizmy uwierzytelniania i autoryzacji (OAuth 2.1), konteneryzację oraz praktyki wytwarzania oprogramowania obejmujące testy integracyjne i testy end-to-end. Efektem jest umiejętność dostarczenia działającej aplikacji wraz z automatyzacją uruchamiania i testowania.

\subsection*{\textit{Wersja w języku angielskim (English version)}}

\textit{The aim of the course is to acquire practical skills in designing and implementing modern web applications divided into backend and frontend layers, based on Domain-Driven Design (DDD) and hexagonal architecture. Students learn the basics of building SOA-style solutions, authentication and authorization mechanisms (OAuth 2.1), containerization, and software engineering practices including integration and end-to-end testing. The intended outcome is the ability to deliver a working application together with automated setup and testing.}

\section{Treści programowe}

\begin{enumerate}
  \item Wprowadzenie do aplikacji internetowych: backend vs frontend, kontrakty API, podstawy jakości i utrzymania.
  \item DDD w praktyce: język ubiquitous, model domenowy, agregaty, invariants, repozytoria, domena vs aplikacja vs infrastruktura.
  \item Architektura oktogonalna: porty i adaptery, granice odpowiedzialności, kierunek zależności, testowalność.
  \item Wstęp do SOA: rozbicie na usługi, kontrakty, wersjonowanie API, spójność danych i integracja usług.
  \item Bezpieczeństwo i autoryzacja: wprowadzenie do OAuth 2.1, tokeny, scenariusze integracyjne (w zakresie kursu).
  \item Konteneryzacja: Docker, obrazy, sieci, wolumeny, docker-compose; uruchamianie pełnego środowiska projektu.
  \item Budowa backendu: warstwy, API, walidacja, obsługa błędów, persystencja danych; integracja z innymi usługami (wg programu).
  \item Budowa frontendu: podstawy architektury aplikacji klienckiej, integracja z API, zarządzanie stanem, obsługa błędów.
  \item Testy integracyjne: strategia testów, testy kontraktów/warstw, uruchamianie z zależnościami (np. kontenery).
  \item Testy end-to-end: scenariusze użytkownika, automatyzacja uruchomienia środowiska i testów.
  \item Elementy procesu wytwarzania: Git flow / PR, code review, podstawy CI (pipeline uruchamiający testy).
  \item Projekt zaliczeniowy: iteracyjna realizacja funkcji, dopięcie architektury, testów i uruchamiania w kontenerach.
\end{enumerate}

\section{Przedmiotowe efekty uczenia się}

\subsection*{}
\begin{itemize}
  \item Student zna i rozumie podstawowe pojęcia DDD (m.in. model domenowy, encje, wartości, agregaty, repozytoria, bounded context) oraz potrafi uzasadnić ich użycie.
  \item Student zna i rozumie założenia architektury oktogonalnej oraz zasady separacji warstw i zależności w systemie.
  \item Student zna i rozumie podstawy SOA oraz typowe style integracji usług (API, komunikacja synchroniczna/asynchroniczna – zależnie od programu zajęć).
  \item Student zna i rozumie mechanizmy uwierzytelniania i autoryzacji w aplikacjach internetowych, w tym OAuth 2.1 na poziomie koncepcji i przepływów.
  \item Student zna i rozumie rolę testów integracyjnych i end-to-end w cyklu wytwarzania oraz podstawy automatyzacji w CI.
\end{itemize}

\subsection*{Umiejętności}
\begin{itemize}
  \item Student potrafi zaprojektować i zaimplementować backend zgodny z DDD i architekturą oktogonalną, z czytelnym podziałem na warstwy oraz granice kontekstów.
  \item Student potrafi przygotować i zintegrować frontend z backendem poprzez API oraz zadbać o podstawowe aspekty bezpieczeństwa sesji/tokenów.
  \item Student potrafi zastosować OAuth 2.1 w aplikacji (konfiguracja i integracja po stronie aplikacji – zgodnie z założeniami kursu).
  \item Student potrafi przygotować środowisko uruchomieniowe w kontenerach (np. Docker + Compose), w tym zależności typu baza danych i usługi pomocnicze.
  \item Student potrafi napisać testy integracyjne dla kluczowych przypadków oraz testy E2E dla scenariuszy użytkownika i uruchamiać je automatycznie.
\end{itemize}

\subsection*{Kompetencje społeczne}
\begin{itemize}
  \item Student jest gotów do pracy zespołowej nad produktem programistycznym, w tym do komunikacji, podziału zadań i przeglądu kodu.
  \item Student jest gotów do odpowiedzialnego podejścia do jakości: testowania, utrzymania spójności architektury i dokumentowania rozwiązań.
\end{itemize}

\section{Kryteria oceny}

\begin{itemize}
  \item Kolokwium (część wykładowa): pytania problemowe + zadania dotyczące DDD, architektury oktogonalnej, SOA oraz OAuth 2.1.
  \item Praca na zajęciach (laboratoria): realizacja zadań cząstkowych, jakość kodu, poprawność architektury, terminowość.
  \item Projekt: kompletność funkcjonalna, zgodność z założeniami (DDD + architektura oktogonalna), jakość testów (integracyjne + E2E), konteneryzacja i uruchamianie środowiska.
  \item Ocena końcowa = składowa z kolokwium + składowa z laboratoriów (praca) + składowa z projektu.
  \item Warunek zaliczenia: oddanie projektu oraz uzyskanie wyniku pozytywnego z kolokwium.
\end{itemize}
